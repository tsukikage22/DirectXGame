# Direct3D 12 自作レンダリングエンジン

Windows 11 / Direct3D 12 上で、レンダリングパイプライン・GPUリソース管理・シーン管理を自作している個人プロジェクトです。
物理ベース（測光量）ライティング、HDRレンダリング、世代付きハンドルによるオブジェクト管理などを実装しています。

> 学習と作品制作を兼ねたプロジェクトで、現在も開発を継続しています。

## 主な技術的特徴

| 分野                 | 内容                                                                                         |
| -------------------- | -------------------------------------------------------------------------------------------- |
| **描画パイプライン** | D3D12 / DXGI 1.6 / フリップモデル、HDR対応（`R16G16B16A16_FLOAT`）、トーンマッピング         |
| **ライティング**     | 物理量（光束 lm・照度 lx・光度 cd）ベースの照明、IESプロファイル対応の測光ライト             |

| **アーキテクチャ**   | App / Engine / Game の3層分離、BuilderパターンによるPSO・ルートシグネチャ構築                |
| **アセット**         | Assimp による GLB/glTF 読み込み、DirectXTex によるテクスチャ処理                             |

すべて C++17 / 生の Direct3D 12 API 上に構築しており、描画・同期・メモリ管理を自前で制御しています。

## 設計上の工夫

### 1. 世代付きハンドルと SlotMap によるオブジェクト管理

GameObject やモデルを生ポインタではなく **`{index, generation}` の世代付きハンドル**で参照します。
削除されたスロットが再利用されても generation が一致しないため、**解放済みオブジェクトへのアクセスを検出**でき、
ダングリングポインタを構造的に防ぎます。

内部の `SlotMap` は、実データを密な配列で保持しつつ間接参照テーブルとフリーリストを持つことで、
**O(1) の生成・削除**と**キャッシュ効率の良い一括走査**を両立しています。

### 2. RAII による D3D12 リソース管理

D3D12 はリソースの寿命管理をアプリケーション側に委ねるため、解放タイミングの誤りが致命的なバグになります。
本プロジェクトでは以下を RAII で管理し、明示的な解放漏れを排除しています。

- **`DescriptorAllocation`** — ディスクリプタヒープのスロットをスコープで確保・返却
- **`ModelLoadScope`** — `ResourceUploadBatch` の Begin/End とアップロードヒープ破棄を一括管理
- **`RetireQueue`** — GPU 実行中のリソースを即座に破棄せず、フレームを跨いで**遅延解放**することで、
  描画に使用中の GPU リソースの破壊を防止

### 3. 物理量（測光）ベースのライティング

明るさを光束 [lm]・照度 [lx]・光度 [cd]といった実世界の測光量で扱います。
平行光源・点光源・スポットライト・測光（IES）ライトの4種を切り替えて描画でき、
IES プロファイル（実在の照明器具の配光データ）をパースして光度テクスチャに変換し、
現実の照明器具に近い配光を再現します。

### 4. Builder パターンによるパイプライン構築

冗長になりがちな `D3D12_GRAPHICS_PIPELINE_STATE_DESC` やルートシグネチャの構築を、
`GraphicsPipelineBuilder` / `RootSignatureBuilder` のメソッドチェーンで宣言的に記述できるようにしています。
